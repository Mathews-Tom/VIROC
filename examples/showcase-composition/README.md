# Showcase composition grammar

Showcase grammar. Non-row composition. Above-floor primitives with explicit Manim degradation.

This is the grammar proof: where `examples/rag-pipeline` is the boring linear `pipeline` flow, this example exercises the `showcase` grammar — panels, code cards, callouts, evidence blocks, fan-out, and comparison compositions arranged in non-row layouts. It walks the same **guided VIROC flow** end to end (concept → script → scene plan → editable VidIR → review → compile → render). `pipeline` stays the boring left-to-right grammar; `showcase` is a separate, bounded grammar — it never extends or special-cases `pipeline`.

## Guided flow

Every committed artifact below is regenerated by the documented commands; the flow is demonstrated, not narrated:

```bash
viroc ingest examples/showcase-composition/authoring-request.yaml   # concept -> authoring-brief.yaml
viroc plan     examples/showcase-composition                         # brief -> script.md, scene-plan.yaml, storyboard.vidir.yaml
viroc critique examples/showcase-composition                         # storyboard -> expected/review/ (script, scene cards, manifest)
viroc compile  examples/showcase-composition --backend static_storyboard
viroc compile  examples/showcase-composition --backend html          # full code/formula fidelity
viroc render   examples/showcase-composition --backend static_storyboard
```

`authoring-request.yaml` is the only hand-authored input (the concept). `authoring-brief.yaml`, `script.md`, `scene-plan.yaml`, and `storyboard.vidir.yaml` are the deterministic `ingest`/`plan` projection of it; `storyboard.vidir.yaml` is the editable VidIR you own. Re-running `ingest`/`plan` is idempotent and byte-stable.

## Scene arc (guided flow)

| Scene | Template | Claim |
|---|---|---|
| `title_card` | statement | The showcase grammar composes content, not just boxes — a title card and a centered claim, no boxes. |
| `primitives` | grid | One of each composition kind in a non-row 2x2 grid: a panel (`data_source`), a code card (`intermediate` → `code`), a callout (`user`), and an evidence block (`storage` → `formula`). |
| `fanout` | fan-out | One resolver service fans out into Concrete IR, a review surface, and a build manifest. |
| `comparison` | comparison | Two portable backend paths compared side by side: HTML versus Remotion source. |

Each composition kind lowers to a Concrete IR primitive: panels and callouts to `rect`, code cards to `code`, evidence blocks to `formula`.

## Top-three parity

Every top-three backend renders the common visual floor (`rect`/`text`/`arrow`) natively. This example also exercises the above-floor `code` and `formula` primitives: HTML and Remotion render them with full fidelity, while Manim renders each as its floor primitive (`rect`) — a deterministic degradation that keeps every object's placement and title and is surfaced as a non-blocking `VIR5033` note, never silently omitted. `viroc compile … --backend manim` therefore exits 0 and prints one `VIR5033` note per degraded object (here: `code` and `formula`); `--backend html` and `--backend remotion` compile with no notes. `static_storyboard` is the review surface and renders the full primitive set.
